# Description

**RAG** is a project aimed at enabling us to create an AI that isn't trained but responds by retrieving knowledge from other sources.

### Goal

Our program must first efficiently index the data from the external source.

Next, it must retrieve the relevant information from that index to answer a given question.

Finally, it must generate a response.

### Overview

When creating an AI model, one of the first steps is to train it. You want the model to
develop skills such as language understanding, reasoning, and structural analysis, and to achieve this, you feed it a huge amount of data.

After training, the model “remembers” what it has learned, but it only “knows” the data it has been given. If you
want it to have more recent knowledge, you must retrain it — a process that takes a long
time.

Training is a technique, and RAG is another one. Instead of feeding the model data
directly, RAG gives the model access to an external source of information and that source
is of your choice.

The two techniques can be combined: the model must still be trained on the key concepts
we’ve mentioned before (language understanding, reasoning, and structural analysis) to
build its foundation, but for knowledge, it can combine its trained data with the external
connection.

### Usage

You have to use `make run ARGS="..."` command.

In the ARGS value you have a choice of not four, not even five, but six options!

**index:**

This commande will read all files you provide in `data/raw` directory.

It will intelligently truncate them into chunks before indexing and store the result to make it searchable later on.

The syntaxe to use it is `make run ARGS="index"`.
You can add the flag `--max_chunk_size` like this `make run ARGS="index --max_chunk_size <number>"` to specify the max chunk's size.
By default it is set to 2000 characters.

You MUST begin by this command before whatever else. Without data ingestion, it's impossible for the model to find answers to your questions.

**search:**

This command allows you to ask one question to the model.

It won't give you the answer to your question, but it will extract (and save in a file) the most relevant chunks that can be used to answer your question.

The syntaxe to use it is `make run ARGS="search --question <'your_question_between_quotes'> --k <number> --save_directory <path_to_the save_directory>"`

`--k` will specify how much relevant chunks you want to retrieve
`--save_directory` will specify iin which directory you want the program store the save file.

**search_dataset:**

This command will do almost the same as above but for a list of multiple questions.

The syntaxe to use it is `make run ARGS="search_dataset --dataset_path <path_to_the_file_which_contain_questions> --k <number> --save_directory <path_to_the save_directory>"`

`--dataset_path` is a flag to specify the path to the file which contain the questions.
**It MUST be a valid JSON file.**

The other flags are the same as above.

**evaluate:**

This command acts as a wrapper for the official `moulinette` binary to evaluate your results against the ground truth dataset. It supports different evaluation modes.

The syntax to use it is: 
`make run ARGS="evaluate <mode> <student_answer_path> <dataset_path> [--k <number>] [--max_context <number>]"`

**Modes (`<mode>`):**
- `search` : Evaluates the global performance of the retrieval step and calculates Recall@k metrics.
- `list` : Inspects retrieval results question by question to help debugging (outputs [VALID] or [INVALID]).

**Arguments:**
- `<student_answer_path>` : The path to the JSON file previously generated by the `search_dataset` command.
- `<dataset_path>` : The path to the ground truth dataset file used to generate your results.
- `--k` and `--max_context` : The number of relevant chunks kept and their maximum size. **These parameters must strictly match the values used during the `search_dataset` execution; otherwise, the evaluation will be skewed.**

**answer:**

The syntaxe to use it is `make run ARGS="answer --question <'your_question_between_quotes'> --k <number>"`

Here, the `--k` flag is optionnal and is set to 10 if you don't pass it.

The result will be automatically store in `data/output/answer_results/single_answer.json`

The result will be a dictionnary with the question's id, the question, a list of the most interesting chunks used to answer the question and the answer.

**answer_dataset:**

The syntaxe to use it is `make run ARGS="answer_dataset --student_search_results_path <path_to_the_save_file_generated_with_search_dataset> --save_directory <path_to_the save_directory>"`

This command will read the file generated by the search_dataset command and answer the questions using the list of selected chunks.

The result will be a list of dictionaries as described in the section on `answer` command.

#### It's not mandatory to specify the names of the flags you want to use (--<flag_name>), but if you do, you must follow the order in which they should be placed.

# System architecture

## Indexing

The `walk` function in the `os` library lets you iterate through all the files and folders starting from a given location.

This allows me to scan through them all and filter out everything except .py, .md, and .txt files.

The indexing process uses the `bm25s` library.

#### Chunking strategy:

`.md` and `.txt` files are indexed as-is, within the chunk size limit.

`.py` files are split into nodes based on their components (functions, classes, comments, etc.) using the ast library.
I create a chunk for each function and class in the file. If there are elements between these nodes, they are concatenated and placed in a chunk, respecting the maximum chunk size limit.

## Retrieval

The retrieval process uses the `bm25s` library.
User queries are tokenized and compared against the indexed chunks using the BM25 algorithm, which scores documents based on exact word matching and term frequency. The system extracts the top `k` most relevant chunks.

## Generation

The text generation step relies on a local LLM accessed via the `ollama` HTTP client (using the `qwen3:0.6b` model).
A strict system prompt is used to restrict the model's behavior, instructing it to act purely as an extraction tool. The model is forced to generate an answer based exactly on the provided context, which is constructed by concatenating the text from the top 5 retrieved chunks.

## Performance analysis

- **Recall@k:** This metric evaluates the retrieval component's ability to rank the chunk containing the correct answer within the top *k* results. Since the system relies on a purely lexical search (BM25), the recall is optimal when queries use the exact terminology found in the source documents. The absence of semantic embeddings or query expansion limits the recall for queries relying on synonyms.

- **System performance & Bottlenecks:** The initial processing time is approximately 3 minutes per 100 questions. Hardware analysis shows that the retrieval step (BM25) is nearly instantaneous; the performance bottleneck is entirely bound by the text generation phase processed by the local LLM via Ollama.

- **Caching Impact:** The implementation of the query caching system drastically improves performance for redundant evaluations, reducing the processing time of 100 cached questions from ~3 minutes to less than 1 second.

## Design decisions

- **Logical Code Chunking (AST):** Instead of relying on an arbitrary character count to split Python files, the system uses the standard `ast` module. This ensures semantic integrity by keeping functions and classes intact, preventing logic from being severed across multiple chunks and providing the LLM with readable context.

- **Lexical vs Vector Retrieval:** The choice of the `bm25s` library ensures extremely fast, CPU-bound indexation with a low memory footprint. It avoids the hardware requirements (GPU) associated with calculating and storing semantic embeddings.

- **Deterministic Generation (LLM Control):** The `qwen3:0.6b` model is configured with a temperature of `0.0` to force deterministic outputs. The strict system prompt confines the model to an extraction role, effectively mitigating algorithmic hallucinations.

- **I/O Optimization (Cache):** Disk read/write operations for the cache are strictly isolated outside the evaluation loop. Using `json.load` at startup and `json.dump` at completion ensures that all cache lookups during execution occur in RAM, preventing disk I/O bottlenecks.

# Additional Features

## Query Caching

To optimize performance and avoid redundant LLM inferences, a query caching mechanism is implemented.

- **Mechanism:** When a question is evaluated, the system generates a unique key based on the question string and the `k` parameter. 
- **Storage:** Results are loaded into an in-memory dictionary during execution and persistently saved to `data/cache/cache.json` at the end of the process.
- **Performance:** Before querying the LLM, the system checks this cache. A cache hit returns the saved answer instantly, bypassing the generation step and reducing processing time from several minutes for a dataset to less than a second.
